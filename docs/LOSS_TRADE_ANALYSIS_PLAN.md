- 市場体制 (RANGING/TRENDING_UP/DOWN)
- 体制信頼度 (0-1)
- ADX値
- ボラティリティ値

**結果情報:**
- 利益/損失 (USD/%)
- 最大ドローダウン (USD/%)
- 保有時間
- 決済理由 (STOP_LOSS/SIGNAL_REVERSAL/EXIT_STRATEGY)

```

├─ 【分類 2: 市場体制】
│  ├─ RANGING での損失
│  ├─ TRENDING での損失
│  └─ 体制判定エラー

├─ 【分類 3: 根本原因】
│  ├─ Type-A: 「フィルターが間違い」
│  │   └─ ✗ PVO閾値が低い → ノイズを拾っている
│  │   └─ ✗ ADX閾値が低い → トレンドがないのにエントリー
│  │   └─ ✗ Volume/Volatility → 取引高が低い時間帯
│  │
│  ├─ Type-B: 「市場体制判定が間違い」
│  │   └─ ✗ RANGING判定なのに方向性あり
│  │   └─ ✗ TRENDING判定なのにレンジ相場
│  │
│  ├─ Type-C: 「シグナルが矛盾している」
│  │   └─ ✗ Strategy が Donchian と逆方向
│  │   └─ ✗ Strategy が複数で相互矛盾
│  │
│  ├─ Type-D: 「市場状況の急変」
│  │   └─ ✗ トレンド反転が急激
│  │   └─ ✗ 経済指標発表直後
│  │
│  └─ Type-E: 「タイミング問題」
│      └─ ✗ Donchian上限で買い、すぐ反転
│      └─ ✗ ボラティリティスパイク直後
```

### 結論と今後の方針

1. **Phase 3 の「Volatility 完全失敗」は誤りの可能性**
   - 生ログ抽出時の値読み違い、またはフィルタリングロジックの相違が原因と推定
   - Trade Log では Volatility FAIL 環境でむしろ高パフォーマンス

   - Phase 3: CI[-263, 947] で有意でない
   - Trade Log: CI[1295, 2207] で有意 ← 新データ基盤が信頼性高い

3. **次フェーズの基盤**
   - すべての Phase 4 分析は **Trade Log JSON** をソースに使用
   - 生ログ抽出による誤りを排除
   - 各バックテスト実行時に自動的に正確なメタデータが記録される

## 利点

1. **根本原因の特定** - 「PVO 閾値が悪い」「RANGING ロジックが弱い」が明確
2. **優先順位付け** - どの改善案から実装すべきかが数字で分かる
3. **統計的正当性** - 勘ではなく、統計で改善案の有効性を判断
4. **勝ちトレード保護** - A/B テストで負の影響を事前に検出
5. **継続改善** - 月次で新しい損失パターンを検出するサイクル
6. **監査可能性** - すべての改善案と効果を記録・追跡可能

## 注意点

- **過剰最適化の回避** - 過去データに過度に最適化しない
- **サンプル数確保** - 最低 100 トレード程度の損失サンプルが必要
