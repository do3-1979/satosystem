# ビットコイン損失トレード分析 - 洗練されたプラン

## 概要

単純な「ドローダウンが高い順に並べて原因特定」ではなく、**システマティックな根本原因分析** と **帰納的な改善検証** を行うフレームワークです。

---

### 1-2. 抽出対象メタデータ

**基本情報:**
- Trade ID, 入場時刻, 決済時刻
- 方向 (BUY/SELL), 入場価格, 決済価格

**条件情報:**
- PVO信号状態
- Donchianブレイク方向
- Strategy評価（A/B/C/なし）
- Strategy と Donchian の一致度

**フィルター情報:**
- 各フィルターの成功/失敗
- フィルター値と閾値

**市場情報:**
- 市場体制 (RANGING/TRENDING_UP/DOWN)
- 体制信頼度 (0-1)
- ADX値
- ボラティリティ値

**結果情報:**
- 利益/損失 (USD/%)
- 最大ドローダウン (USD/%)
- 保有時間
- 決済理由 (STOP_LOSS/SIGNAL_REVERSAL/EXIT_STRATEGY)

## フェーズ 2: 損失トレード分類と根本原因分析

### 2-1. 多次元分類

```
損失トレード
├─ 【分類 1: ドローダウン深さ】
│  ├─ 深刻 (< -3.0%)
│  ├─ 中程度 (-1.0% to -3.0%)
│  └─ 浅い (-0.1% to -1.0%)
│
├─ 【分類 2: 市場体制】
│  ├─ RANGING での損失
│  ├─ TRENDING での損失
│  └─ 体制判定エラー
│
├─ 【分類 3: 根本原因】
│  ├─ Type-A: 「フィルターが間違い」
│  │   └─ ✗ PVO閾値が低い → ノイズを拾っている
│  │   └─ ✗ ADX閾値が低い → トレンドがないのにエントリー
│  │   └─ ✗ Volume/Volatility → 取引高が低い時間帯
│  │
│  ├─ Type-B: 「市場体制判定が間違い」
│  │   └─ ✗ RANGING判定なのに方向性あり
│  │   └─ ✗ TRENDING判定なのにレンジ相場
│  │
│  ├─ Type-C: 「シグナルが矛盾している」
│  │   └─ ✗ Strategy が Donchian と逆方向
│  │   └─ ✗ Strategy が複数で相互矛盾
│  │
│  ├─ Type-D: 「市場状況の急変」
│  │   └─ ✗ トレンド反転が急激
│  │   └─ ✗ 経済指標発表直後
│  │
│  └─ Type-E: 「タイミング問題」
│      └─ ✗ Donchian上限で買い、すぐ反転
│      └─ ✗ ボラティリティスパイク直後
```

### 2-2. パターン検出

```
【質問】同じ条件で繰り返し損失していないか？

例: 以下の条件が重なると常に損失
- Volatility フィルター: ✗ (値が高い状態)
- ADX: 低い (25未満)
- 市場体制: RANGING と判定

→ この条件下では「エントリー禁止」にする
```

### 2-3. 因果関係マトリックス

```
┌─────────────────┬──────┬──────┬────────┬────────┐
│ 条件/フィルター │ 損失 │ 勝ち │ 勝率 % │ 平均PnL │
├─────────────────┼──────┼──────┼────────┼────────┤
│ PVO > 100       │  15  │  45  │ 75.0% │ +520   │
│ PVO 10-100      │  28  │  52  │ 65.0% │ +180   │
│ PVO < 10        │  22  │  8   │ 26.7% │ -340   │
│                 │      │      │        │        │
│ ADX >= 30       │  12  │  58  │ 82.8% │ +680   │
│ ADX 20-30       │  18  │  35  │ 66.0% │ +280   │
│ ADX < 20        │  35  │  12  │ 25.5% │ -420   │
│                 │      │      │        │        │
│ TRENDING        │  22  │  75  │ 77.3% │ +520   │
│ RANGING         │  28  │  25  │ 47.2% │ -180   │
└─────────────────┴──────┴──────┴────────┴────────┘

洞察:
- PVO < 10 で必ず負ける → ロジックが間違っている
- ADX < 20 でも損失が多い → ADX閾値を上げるべき
- RANGING で勝率が低い → レンジ相場ロジックが弱い
```

## フェーズ 3: 有効性検証フレームワーク

### 3-1. 改善案の生成と優先順位付け

```
【改善案 A】PVO 閾値を 10 → 50 に引き上げ
  - 効果期待度: 高 (PVO < 10 での損失が多い)
  - 計算: 実現損失削減 = 〇〇 USD
  - リスク: 取引機会削減 30%
  - ROI: 利益効率 20% 向上

【改善案 B】RANGING でのエントリー禁止
  - 効果期待度: 中 (勝率が大幅改善する可能性)
  - 計算: 損失削減 = △△ USD
  - リスク: 取引機会削減 25%
  - ROI: 勝率改善で補う

【改善案 C】ADX 動的閾値化
  - 市場体制別に ADX 閾値を変更
  - TRENDING: ADX >= 20
  - RANGING: ADX >= 40
  - 効果期待度: 高
```

### 3-2. A/B テスト検証

```
バックテスト期間: 2024-01-01 ~ 2025-12-31 (12ヶ月)

【Control グループ】現在のロジック
- 総利益: +5,200 USD
- 勝率: 62.5%
- ドローダウン: -1,850 USD
- Sharpe: 1.2

【Test グループ】改善案 A 適用
- 総利益: +6,100 USD (+16.3%)
- 勝率: 68.2% (+5.7%)
- ドローダウン: -1,200 USD (-35%)
- Sharpe: 1.6 (+33%)
```

### 3-3. 統計的有意性確認

```
【カイ二乗検定】勝率の改善が統計的に有意か？
- H0: 改善前後で勝率に差なし
- サンプル数: 150トレード以上
- 有意水準: p < 0.05

【ブートストラップ検証】改善の安定性
- 100回のリサンプリング
- 95% 信頼区間での PnL 分布確認
```

## フェーズ 4: ホットテスト前のチェックリスト

### 4-1. 改善案の品質確認

```
□ ドローダウン深刻な損失トレード >= 10 個 分析
□ 損失パターン >= 3 個 特定
□ 各パターンの発生頻度 >= 5% 以上
□ 改善案のバックテスト期間 >= 1 年
□ 統計的有意性 p < 0.05 を達成
□ 勝ち率への負の影響 < 2%
```

### 4-2. リスク評価

```
改善案実施後の懸念:
1. 取引機会削減 → でも利益効率向上で補える？
2. 新パターンの出現 → 監視期間を設定
3. 市場状況変化 → 月次での再検証
```

---

## フェーズ 5: 継続改善サイクル

```
【Step 1】ホットテスト実施（1-2 週間）
  └─ 実際の市場で改善案を検証

【Step 2】結果分析
  └─ バックテストの予測と実績の乖離を分析

【Step 3】微調整
  └─ 予測と異なれば、閾値やロジックを微調整

【Step 4】フリーズ期間
  └─ 確定したロジックで 1 ヶ月走らせる

【Step 5】再びフェーズ 2 に戻る
  └─ 新しい損失パターンが出現しているか確認
```

## 実装ツール

### 必要なスクリプト

1. **trade_extractor.py** - ログからトレード単位で抽出
2. **trade_analyzer.py** - メタデータベース分析
3. **pattern_detector.py** - 損失パターン検出
4. **hypothesis_generator.py** - 改善案自動生成
5. **backtest_validator.py** - 改善案をバックテストで検証

### 出力成果物

- `trades_with_metadata.csv` - トレード全体 (条件/フィルター/結果)
- `loss_trades_analysis.html` - 損失トレード分析レポート（インタラクティブ）
- `pattern_matrix.csv` - 因果関係マトリックス
- `improvement_candidates.json` - 改善案リスト（優先度付き）
- `backtest_results.html` - 改善案の検証結果

---

## 利点

1. **根本原因の特定** - 「PVO 閾値が悪い」「RANGING ロジックが弱い」が明確
2. **優先順位付け** - どの改善案から実装すべきかが数字で分かる
3. **統計的正当性** - 勘ではなく、統計で改善案の有効性を判断
4. **勝ちトレード保護** - A/B テストで負の影響を事前に検出
5. **継続改善** - 月次で新しい損失パターンを検出するサイクル
6. **監査可能性** - すべての改善案と効果を記録・追跡可能

---

## 注意点

- **過剰最適化の回避** - 過去データに過度に最適化しない
- **サンプル数確保** - 最低 100 トレード程度の損失サンプルが必要
- **市場変化への対応** - 季節性や市場レジーム変化を考慮
