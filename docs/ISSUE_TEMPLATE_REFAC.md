# Refactoring / Improvement Issue Template

```
タイトル: [C<ID>] 簡潔な要約 (例: C2 side表記統一 ENUM化)
カテゴリ: C<ID>
背景: 現状の問題点 / 発生頻度 / 影響範囲
ゴール: 成果指標 (性能向上%、保守性、バグ件数減少等)
範囲: 変更対象ファイル / 非対象領域 / 逆影響が出る懸念箇所
アプローチ案: 方針 (軽量パッチ / 抜本的再設計 / 段階的分離)
破壊的変更有無: 有 / 無 (後方互換説明)
テスト計画: 追加/修正テスト一覧 + 検証メトリクス
ロールバック手順: 失敗時の即時復旧方法
完了条件: 受け入れ基準 (例: 全テスト緑 / 指標改善達成) 
関連Issue: #番号 (任意)
``` 

## 優先度基準
1. 安定性 (クラッシュ/誤発注) 影響度
2. 戦略改善阻害度 (複雑性/指標追加困難)
3. 性能ボトルネック (過剰API/計算)
4. 保守性 (循環依存/肥大クラス)

## 例
```
タイトル: [C2] side表記統一 ENUM化
カテゴリ: C2
背景: 'BUY','SELL','NONE','buy','sell','None' が混在し判定条件/ログ集計で分岐増大。
ゴール: 内部表記を Enum Side に一本化し交換境界のみ変換。関連コード分岐数30%削減。
範囲: bot.py, trading_strategy.py, risk_management.py, bybit_exchange.py, order.py
アプローチ案: 新規 side_enum.py 追加 / 既存文字列比較を Enum 比較へ置換。
破壊的変更有無: 無 (外部API I/Oは変換ヘルパ対応)
テスト計画: 既存売買フロー回帰 + Enum 不正値時例外テスト
ロールバック手順: Enum導入コミット revert
完了条件: grepで 'BUY'/'SELL' の生比較ゼロ / 全テスト緑
関連Issue: なし
```
