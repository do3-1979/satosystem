## パラメータ最適化と市場適応性の検証仮説

### ユーザーの質問の本質

```
質問1: "アプローチとして、2024年のパラメータを改造して2025年の勝てるようにするということ？"
→ 本来のアプローチ: 「2024最適パラメータ」を修正して「2025に対応」させるのか？

質問2: "2025年に最適化したパラメータを2024年に適用した場合はどうなる？"
→ 逆検証: 2025最適パラメータを時間をさかのぼって2024年に適用したら成績は改善か悪化か？

質問3: "ボラティリティの多寡によってパラメータを変えるという基本的な戦略ですか？"
→ 本質的な質問: 単なる「ボラティリティ適応」なのか、それとも「市場レジーム判定」なのか？
```

### 仮説フレームワーク

#### 仮説A: パラメータ最適化は市場依存的
```
前提: k2/k3の最適値は市場環境に完全に依存する

実験1: 2025最適パラメータを2024に適用
期待値: 2024年パフォーマンス悪化（2025用に最適化されているため）

実験2: 2024最適パラメータを2025に適用
結果: 実際に大失敗（観測済み）

結論: パラメータの「通用性」がない = 固定的な最適化は無意味
```

#### 仮説B: 「ボラティリティ適応」は必要条件だが十分条件ではない
```
前提: ATR/ボラティリティが変わるとパラメータも変わるべき

単純化: 低ボラティリティ → k2/k3を狭める
      高ボラティリティ → k2/k3を広げる

問題1: では「最適な狭さ」「最適な広さ」はいくら？
      → ボラティリティレベルごとに最適化が必要

問題2: ボラティリティと「トレンド強度」は別物
      → 低ボラティリティでも強いトレンドはある
      → 高ボラティリティでもノイズだけのこともある

結論: 単なる「ボラティリティ適応」では不十分 = レジーム判定が必須
```

#### 仮説C: 「市場レジーム判定」がコア戦略
```
前提: 2024年と2025年は「全く異なる市場レジーム」

レジーム分類:
- STRONG_TREND: トレンド明確 (2024年1-3月, 6-7月)
- WEAK_TREND: トレンド弱い (2025年1-5月)
- SIDEWAYS: トレンドなし (2025年6-10月)

各レジームで最適パラメータは異なる
→ 時系列で見ると「その時の市場レジーム用に再最適化」が必要

結論: 「固定パラメータ」ではなく「動的パラメータ + レジーム判定」が必須
```

### 検証計画

#### Step 1: 2025年最適パラメータを逆算
```
方法: グリッドサーチまたは最適化ツール
範囲: k2 = 1.2 ~ 2.5, k3 = 1.0 ~ 2.0
目標: 2025年(1-10月)で最高のシャープレシオを実現するk2/k3を特定
```

#### Step 2: 交差検証テスト
```
Test A: (2025最適k2/k3) を 2024年に適用
        → 2024年の成績は改善か悪化か？

Test B: (2024最適k2/k3) を 2025年に適用
        → すでに実測: 大失敗 (-70%, 1.92%勝率)

クロス結果の解釈:
両者の結果が「逆」なら → パラメータ最適化は市場依存的
両者の結果が「同じ向き」なら → ボラティリティ以外の要因も重要
```

#### Step 3: ボラティリティ相関分析
```
測定項目:
- 2024年各月のATR平均
- 2025年各月のATR平均
- k2/k3との勝率相関
- k2/k3とシャープレシオ相関

仮説検証:
「高ATR月では大きいk値が有効」
「低ATR月では小さいk値が有効」
が本当に成立するか？
```

#### Step 4: レジーム判定の検証
```
方法: 2024/2025の月単位で「その月の市場レジーム」を分類
      STRONG_TREND, WEAK_TREND, SIDEWAYS など

分析:
- STRONG_TREND 月の最適k値は？
- WEAK_TREND 月の最適k値は？
- SIDEWAYS 月の最適k値は？

結果:
レジームごとに「推奨k値セット」を作成
→ これが動的パラメータ調整の根拠になる
```

## 実装方針の3つの選択肢

### オプション1: 単純ボラティリティ適応
```python
def get_dynamic_k2_k3(atr):
    """
    ATRに応じてk2/k3を線形に調整
    """
    if atr > high_threshold:
        return k2=2.5, k3=1.6
    elif atr > mid_threshold:
        return k2=2.2, k3=1.6
    else:
        return k2=1.8, k3=1.4

問題: パラメータの「カットオフ値」をどうやって決定する？
     → 試行錯誤になる可能性が高い
```

### オプション2: レジーム判定+マルチパラメータ
```python
def get_adaptive_k2_k3(market_regime):
    """
    市場レジームごとに最適パラメータを返す
    """
    regime_params = {
        'STRONG_TREND': {'k2': 2.5, 'k3': 1.5},
        'WEAK_TREND': {'k2': 1.8, 'k3': 1.4},
        'SIDEWAYS': {'k2': 1.5, 'k3': 1.2},
        'UNSTABLE': {'k2': 0, 'k3': 0}  # 取引禁止
    }
    return regime_params[market_regime]

メリット: 各レジームで個別最適化可能
デメリット: レジーム判定ロジックの構築が複雑
```

### オプション3: リアルタイム機械学習適応
```python
def adaptive_k2_k3_with_ml():
    """
    過去N営業日のパフォーマンスに基づいて
    k2/k3を動的に調整
    """
    recent_performance = backtest_last_N_days_with_multiple_k_values()
    best_k2, best_k3 = find_best_parameters(recent_performance)
    return best_k2, best_k3

メリット: 常に最新の市場に最適化
デメリット: 過学習リスク、計算コスト大
```
